# CLD — приоритизированный roadmap до публикации в сторы

## Статус
Сейчас у нас есть прототип.
Цель — довести приложение до состояния, пригодного для публикации в App Store и Google Play.

## Главный принцип
Сначала делаем **MVP hardening**, потом **store readiness**, и только потом — premium/growth-фичи.

---

## Что делаем прямо сейчас
Это ближайшие задачи, без которых приложение всё ещё выглядит как прототип.

### 1. Довести базовый user flow до нормального состояния
- [ ] локальное сохранение счётчиков
- [ ] welcome/onboarding только на первом запуске
- [ ] понятный flow: welcome → выбор привычки → создание → список → детали
- [ ] пустое состояние без ощущения “демо”
- [ ] аккуратный экран деталей
- [ ] аккуратная форма создания / редактирования

### 2. Убрать dev-ощущение из приложения
- [ ] production app name
- [ ] production package name / bundle id
- [ ] иконка приложения
- [ ] launch/splash experience
- [ ] единый визуальный стиль
- [ ] привести тексты к продуктовой формулировке

### 3. Подтянуть стабильность
- [ ] отсутствие ошибок в `flutter analyze`
- [ ] стабильный запуск на iOS
- [ ] стабильный запуск на Android
- [ ] проверка create/edit/delete/reset flows
- [ ] проверка сохранения данных после перезапуска

## Продуктовый ориентир
Скрины `clean_day_*` используем как UX/UI референс конкурента для Android и iOS.
Цель — сделать приложение не копией, а более чистой и аккуратной версией по ощущениям.

---

## Правило работы с массовыми правками
Для этого проекта большие изменения нужно отдавать не через inline patch, а через готовый shell-скрипт, который перезаписывает нужные файлы.

### Приоритет форматов
1. shell-скрипт с полными файлами
2. полный готовый файл
3. точечная замена
4. git patch — только если явно нужен и точно проверен
